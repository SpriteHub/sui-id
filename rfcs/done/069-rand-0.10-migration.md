# RFC 069 — rand 0.10 migration

**Status.** Implemented (v0.57.1)
**Priority.** P2 — maintenance; no user-visible change.
**Tracks.** Dependency refresh (cargo outdated audit, v0.57.1 session).
**Touches.** `crates/sui-id-core` (hibp, mfa, tokens, password, setup, admin,
jwt tests), `crates/sui-id-store` (crypto), `crates/sui-id` (backup, csrf,
startup, main). Workspace `Cargo.toml` and `Cargo.lock`.

## Implementation note (v0.57.1)

**Option B chosen** — `getrandom::fill` + `SigningKey::from_bytes` +
`Zeroizing<[u8; 32]>`. Does not require waiting for ed25519-dalek 3.x.

### Changes made

**Workspace `Cargo.toml`:**
- `rand = "0.8"` → `rand = "0.10"`
- `rand_core = { version = "0.6", features = ["std"] }` →
  `rand_core = "0.10"` (the `std` feature was removed in 0.10)
- `getrandom = "0.4"` added as a direct workspace dep
- `zeroize` was already a workspace dep (via ed25519-dalek); no change needed

**`OsRng.fill_bytes` → `getrandom::fill` (10 sites):**
Every call site that used `rand::rngs::OsRng.fill_bytes(&mut buf)` or
`rand_core::OsRng.fill_bytes(...)` was replaced with
`getrandom::fill(&mut buf).expect("system RNG unavailable")`.

Files updated: `forgot_password.rs`, `mfa.rs` (×2), `tokens.rs`,
`password.rs` (via `encode_b64`), `backup.rs` (×3), `csrf.rs`,
`startup.rs`, `main.rs`.

**`SaltString::generate(&mut OsRng)` → `encode_b64` (1 site):**
`password.rs` now generates a 16-byte random salt via `getrandom::fill`,
then encodes it with `SaltString::encode_b64`. Semantically identical
(128-bit random salt); avoids the `rand_core 0.6 CryptoRng` trait bound.

**`SigningKey::generate(&mut rng)` → Option B (2 production sites):**
`setup.rs` and `admin.rs` now use:
```rust
let mut secret = Zeroizing::new([0u8; 32]);
getrandom::fill(secret.as_mut()).expect("system RNG unavailable");
let sk = SigningKey::from_bytes(&secret);
// secret zeroized on drop
```
Cryptographically equivalent. Memory-safe via `Zeroizing<>`.

**JWT tests (4 sites):** `SigningKey::generate(&mut OsRng)` →
`SigningKey::from_bytes(&[1u8; 32])`. Deterministic keys are appropriate
(and preferable) for unit tests.

**`rand_core` removed as direct dep** from `sui-id-core/Cargo.toml`,
`sui-id-store/Cargo.toml`, `sui-id/Cargo.toml`.
`getrandom` added to those three crates.

### Outcome

- `cargo check --workspace` clean.
- **228/228 library tests pass** (175/175 from 5 crates + 53 from `sui-id`).
- Zero references to `OsRng` or `rand_core` in production code.
- `rand_core 0.6` remains in the tree as a transitive dep of
  `ed25519-dalek 2.x` and `password-hash 0.5` — unavoidable until
  those crates update to rand_core 0.9/0.10.

## Background

The rand 0.8 ecosystem (`rand = "0.8"`, `rand_core = "0.6"`) is the last
version that shipped the `rand::rngs::OsRng` type and the
`rand_core::RngCore` trait at their familiar import paths. rand 0.9 and 0.10
restructured the ecosystem significantly:

| Old (rand 0.8 / rand_core 0.6) | New (rand 0.10 / rand_core 0.10) |
|---|---|
| `rand::rngs::OsRng` | Removed — no direct replacement in rand |
| `rand_core::OsRng` | `OsRng` removed from rand_core; use `getrandom::fill()` |
| `rand::RngCore` / `rand_core::RngCore` | `rand_core::Rng` (trait renamed; `RngCore` is a blanket alias) |
| `rand_core = { version = "0.6", features = ["std"] }` | `rand_core = "0.10"` — the `std` feature no longer exists |

The project attempted the upgrade during the v0.57.1 session and found
three categories of call sites, with one category being a blocker.

---

## Blocker: ed25519-dalek 2.x is pinned to rand_core 0.6

`ed25519-dalek = "2.1"` depends on `rand_core = "0.6"`.
`SigningKey::generate(&mut rng)` expects a `CryptoRng + RngCore`
from **rand_core 0.6's** trait definitions.

rand 0.10 uses rand_core 0.10. The two major versions of rand_core
cannot satisfy the same trait bound — `rand_core 0.10::OsRng`
does not implement `rand_core 0.6::CryptoRng`.

`SigningKey::generate` is used in **production code**:
- `sui-id-core/src/setup.rs` — generates the initial signing key during
  the setup wizard.
- `sui-id-core/src/admin.rs` — generates signing keys when an operator
  adds a new key pair via the admin panel.

These are security-critical paths; the keys must be generated with a
cryptographically secure RNG. Using `from_bytes` with a fixed seed
(acceptable for tests) is not acceptable here.

### Resolution options

**Option A — Wait for ed25519-dalek 3.x (recommended).**
ed25519-dalek 3.0.0-pre.7 is already in development and targets
rand_core 0.9+. When a stable ed25519-dalek 3.x ships, this RFC
can land alongside it by updating both constraints in a single pass.
This avoids a split migration.

**Option B — Replace `SigningKey::generate` with `getrandom` + `from_bytes`.**
```rust
// Replaces: SigningKey::generate(&mut OsRng)
let mut secret = [0u8; 32];
getrandom::fill(&mut secret).expect("system RNG unavailable");
let sk = SigningKey::from_bytes(&secret);
// Zeroize secret after use:
secret.zeroize();
```
This is cryptographically equivalent (the key material still comes from
the OS RNG), removes the ed25519-dalek rand_core dependency entirely,
and lets the upgrade proceed now. Requires adding `zeroize` as a
dependency and auditing that the compiler does not optimize out the
zeroize call (use `zeroize::Zeroize` or `zeroize::Zeroizing`).

**Option C — Keep rand 0.8 for the ed25519-dalek call sites only, add
`rand = "0.10"` alongside it.**
Cargo allows both rand 0.8 and rand 0.10 in the dep tree, with the
workspace using rand 0.10 for new code and retaining rand 0.8 for the
`SigningKey::generate` calls via explicit path
(`rand_08::rngs::OsRng`). This is fragile and confusing — not recommended.

**Recommendation: Option A unless ed25519-dalek 3 is not imminent, in
which case Option B.**

## Non-blocking call sites (straightforward to migrate)

These all use `OsRng.fill_bytes(&mut buf)` — simple random byte generation
that does not involve the ed25519 trait bounds.

| File | Pattern | Migration |
|---|---|---|
| `sui-id-core/src/forgot_password.rs` | `OsRng.fill_bytes(&mut bytes)` | `getrandom::fill(&mut bytes).expect("rng")` |
| `sui-id-core/src/mfa.rs` (×2) | `OsRng.fill_bytes(&mut secret/bytes)` | Same |
| `sui-id-core/src/tokens.rs` | `OsRng.fill_bytes(&mut buf)` | Same |
| `sui-id-store/src/crypto.rs` (×2) | `OsRng.fill_bytes(&mut k/nonce)` | Same |
| `sui-id/src/backup.rs` (×3) | `OsRng.fill_bytes(&mut salt/nonce/byte)` | Same |
| `sui-id/src/csrf.rs` | `OsRng.fill_bytes(&mut bytes)` | Same |
| `sui-id/src/startup.rs` | `OsRng.fill_bytes(...)` | Same |
| `sui-id/src/main.rs` | `rand::rngs::OsRng.fill_bytes(...)` | Same |

**`password.rs` — `SaltString::generate(&mut OsRng)`:**
`argon2` / `password-hash` use `rand_core 0.6`'s `CryptoRng` trait for
salt generation. This is the same ed25519-dalek-class problem. Options:
- Wait for password-hash crates to update to rand_core 0.9+.
- Or replace with `SaltString::from_b64(base64_of_getrandom_12_bytes)` —
  valid but slightly more verbose.

**JWT tests — `SigningKey::generate(&mut OsRng)`:**
Test-only; not security-critical. Already demonstrated as safe to replace
with `SigningKey::from_bytes(&[1u8; 32])` (deterministic, appropriate for
unit tests). This part can land immediately, independent of the blocker.

## Workspace changes required

```toml
# Cargo.toml workspace section — before
rand      = "0.8"
rand_core = { version = "0.6", features = ["std"] }

# After
rand      = "0.10"
rand_core = "0.10"          # std feature removed; always available
getrandom = "0.4"           # new direct dep for fill_bytes replacements
```

The `rand_core` dep on crate Cargo.toml files that used it only for
`OsRng.fill_bytes` should be replaced with `getrandom`.

  `getrandom::fill(...).expect("system RNG unavailable")`.
- [ ] `SaltString` generation uses a rand_core 0.10-compatible path.

  or Option B above).

- [ ] No regression to `inline-style-bound`, `text-leaks`, or
  `semantic-parity` invariants (those are unaffected, but the CI gate
  should still be confirmed green).
